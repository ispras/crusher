
# Общая информация

В данном примере представлен пайплайн анализа `OpenSSL` сервера: от подготовки окружения до анализа результатов фаззинга.

Фаззинг-цель - сервер OpenSSL v1.1.1j.

Для автоматизации фаззинга применяется компилятор `ISP-Circea`, который позволяет:

- автоматически учитывать состояния сервера;
- автоматически создавать генератор входных данных на базе клиента.

Демонстрируется воспроизведение известной ошибки CVE-2021-3449, которая проявляется в следующем сценарии:

1) Установка TLS-соединения;
2) Отправка невалидного зашифрованного сообщения `ClientHello` в рамках пересогласования параметров TLS-соединения.

План анализа:

1. Подготовка окружения;
2. Подготовка OpenSSL-сервера (объект фаззинга);
3. Подготовка мод. OpenSSL-клиента (генератор вх. данных);
4. Подготовка (валидного) сценария взаимодействия клиента с сервером;
5. Генерация нач. вх. данных на эталонной сессии;
6. Создание вспомогательных скриптов для фаззинга;
7. Запуск фаззинга;
8. Анализ результатов:

    1) воспроизведение крешей;
    2) покрытие по исходному коду:
        - LCOV: строки, функции
        - [MC/DC](https://en.wikipedia.org/wiki/Modified_condition/decision_coverage)
    3) покрытие по бинарному коду (для [Lighthouse](https://github.com/gaasedelen/lighthouse))

См. также разделы "ISP-Circea", "Фаззинг протоколов с использованием модифицированного клиента" и "Дифференциальный фаззинг" в Руководстве пользователя (`crusher/README.pdf`).

# 1. Подготовка окружения

Анализ будем проводить в докер-контейнере на основе образа `openssl-circea:latest`.

1. Сборка докер-образа `openssl-circea`:
```shell
$ ./docker/docker_build.sh
``` 

2. Создание контейнера (укажите актуальный путь до директории `crusher`):
```shell
$ ./docker/docker_run.sh <crusher_dir> [hasp_ip]
```
где:

- `crusher_dir` - директория `crusher/` (обязательная опция)
- `hasp_ip` - IP сервера лицензий (необязательная опция)

# 2. Подготовка OpenSSL-сервера

Подготовка OpenSSL-сервера включает следующие шаги:

1. Оптимизация работы сервера;
2. Создание нескольких сборок сервера с разными типами инструментации:

- основная сборка (с битовой картой покрытия)
- asan-сборка (включен address-санитайзер)
- msan-сборка (включен memory-санитайзер)
- lcov-сборка - для отчёта о покрытии исходного кода
- чистая сборка (без инструментации) - для отчёта о покрытии бинарного кода

3. Создание SSL-сертификата и ключа (уже готовы)

## 2.1. Оптимизация работы сервера

Все патчи - см. `server/server.patch`. Далее разберём каждый фрагмент детально.

### 2.1.1. Дерандомизация

Для детерминированной работы сервера необходимо вызывать ф-цию `FuzzerSetRand` (см. https://github.com/profuzzbench/profuzzbench/blob/master/subjects/TLS/OpenSSL/rand.patch)
в начале каждого дочернего процесса fork-сервера.

### 2.1.2. Установка точки вызова fork-сервера

Точку fork-а необходимо поставить как можно позже в программе, но до того момента, как начинают обрабатываться вх. данные.
В серверах можно отслеживать вызовы `select` и `accept`.
В данном примере перед вызовом `do_server` (`s_server.c`) необходимо вставить следующий код:
```c
    #ifdef __AFL_HAVE_MANUAL_CONTROL
        __AFL_INIT();
    #endif
```

## 2.2. Создание нескольких сборок сервера с разными типами инструментации

Скачивание исходников, применение патчей и выполнение сборок:
```shell
$ ./server/build.sh
```

Для автоматического получения обратной связи по состояниям сервера используются следующие флаги `ISP-Circea`:

- `-circea-recursion`
- `-circea-state-var`
- `-circea-send`
- `-circea-array-indexing`

Также в этом скрипте создаются SSL-сертификат и ключ.

## 2.3. Создание SSL-сертификата и ключа

```shell
$ mkdir server/keys
$ ./server/openssl-fuzz req -newkey rsa:2048 -new -nodes -x509 -days 3650 \
  -keyout server/keys/key.pem -out server/keys/cert.pem -batch \
  -config server/openssl/apps/openssl.cnf
```

# 3. Подготовка мод. OpenSSL-клиента

Подготовка OpenSSL-клиента включает следующие шаги:

1. Оптимизация работы клиента;
2. Создание генератора вх. данных для сервера на основе клиента с помощью `ISP-Circea`.

## 3.1. Оптимизация работы клиента

Все патчи - см. `client/client.patch`. Далее разберём каждый фрагмент детально.

### 3.1.1. Дерандомизация

Аналогично серверу.

### 3.1.2. Завершение процесса по таймауту

В процессе взаимодействия как сервер, так и клиент могут зависнуть.
Чтобы избежать этого (= ускорить фаззинг), необходимо ограничить время выполнения процесса.

Для этого устанавливается обработчик на сигнал SIGALRM в `s_client.c`:
```c
#include <signal.h>

void sig_handler(int signum){
  printf("SIGALARM handler - exit\n");
  exit(0);
}

int s_client_main(int argc, char **argv)
{
    signal(SIGALRM, sig_handler);
    ...
}
```

и выставляется таймер перед вызовом `init_client`:
```c
    if (getenv("FUZZ")) {
      ualarm(100000, 25000);
    }

    if (init_client(&sock, host, port, bindhost, bindport, socket_family,
                    socket_type, protocol, tfo, !isquic, &peer_addr) == 0) {
        BIO_printf(bio_err, "connect:errno=%d\n", get_last_socket_error());
        BIO_closesocket(sock);
        goto end;
    }
```

### 3.1.3. Добавление мутаций

В ф-ции `put_value` (`packet.c`) выполняется запись целочисленного значения `value` в буфер.
Работа с данными в этой ф-ции выполняется ещё до шифрования.
Сделаем в начале неё мутирующую вставку:
```c
    value = mutate_int(value, len);
```

Также необходимо добавить соответствующий include:
```c
    #include "/opt/crusher/circea/include/mod-client-api.h"
```

## 3.2. Создание генератора вх. данных для сервера на основе клиента с помощью `ISP-Circea`

Скачивание исходников, применение патчей и выполнение сборки:
```shell
$ ./client/build.sh
```

В процессе сборки выполняется автоматическая модификация клиента, а именно добавляются мутации для переменных состояния (флаг `-circea-state-var-mutate`).
Подробнее - см. "Фаззинг протоколов с использованием модифицированного клиента" в документации.

# 4. Подготовка (валидного) сценария взаимодействия клиента с сервером

1. Протокол взаимодействия.

TLS 1.2 (указывается в опциях сервера и клиента)

2. Сервер использует сгенерированные заранее rsa-ключ и x509-сертификат.

Указываются в опциях сервера.

3. Сервер завершается после завершения 1-го tcp-соединения.

Указывается в опциях сервера.

4. Нач. сценарий взаимодействия.

TLS-рукопожатие (handshake) + переустановка параметров TLS-сессии (renegotiation) + завершение TLS-соединения.

По умолчанию при запуске openssl-сервера (`openssl s_server`) и клиента происходит установление TLS-соединения,
после чего можно в stdin писать сообщения.

Также можно передавать команды на пересогласование параметров соединения (`R`) и его завершение (`Q`).

В код клиента были добавлены доп. патчи (`s_client.c`), чтобы команды `R` и `Q` выполнялись сразу после TLS-рукопожатия:
```c
// патч - вспомогательные переменные
uint8_t hs_done = 0;
uint8_t reneg_start = 0;
uint8_t reneg_done = 0;
...

print_stuff(bio_c_out, con, full_log);
hs_done = 1; // патч - рукопожатие выполнено
...

static int user_data_process(struct user_data_st *user_data, size_t *len,
size_t *off)
{

    // патч - автоматическое выполнение renegotiation- и quit-команд
    if (hs_done && !reneg_done) {
        reneg_done = 1;
        return user_data_execute(user_data, USER_COMMAND_RENEGOTIATE, NULL);
    } if (reneg_done) {
        return user_data_execute(user_data, USER_COMMAND_QUIT, NULL);
    }
    
    ...
}
```

В итоге сервер будем запускать такой командой (пути и порт могут меняться):

```shell
$ ./server.sh
```

а клиент такой:

```shell
$ ./client.sh
```

# 5. Генерация нач. вх. данных

1) Создать папку `in`:
```shell
$ mkdir in
```

2) Запустить сервер:
```shell
$ ./server.sh
```

3) Запустить мод. клиент:
```shell
$ GEN_FILE=$PWD/in/session ./client.sh
```

В переменной окружения `GEN_FILE` указывается файл, куда будет записан образец нач. входных данных.

Обратите внимание, что при добавлении/удалении вызовов `mutate_*`, а также при изменении мутирующих circea-* флагов компиляции для клиента необходимо пересоздать начальный образец вх. данных.

# 6. Создание вспомогательных скриптов для фаззинга

## 6.1. Скрипт запуска мод. клиента

См. `plugins/client.py`:
```python
...
    # Здесь необходимо указать команду запуска клиента.
    # Используется значение ранее полученного порта.
    command = [f"{target_dir}/client/openssl-clean",
               "s_client", "-tls1_2",
               "-connect", f"127.0.0.1:{str(port)}",
               "-legacy_renegotiation"]
    # Запуск процесса клиента 
    # (в некоторых случаях может потребоваться указать stdin=None).
    client = Popen(command, stdin=PIPE)
...
```

В большинстве случаев в этом скрипте необходимо менять только команду запуска клиента
и в некоторых случаях - значание параметра `stdin` при создании процесса клиента с помощью вызова `Popen`.

## 6.2. Скрипт модификации опций

См. `plugins/conf.py`.

В методе `transform_options` бинарь openssl-сервера заменяется на asan- и msan-сборки в отдельных fuzz-процессах - для обеспечения дифференциального фаззинга.

См. также раздел "Пользовательские скрипты модификации опций" в документации.

# 7. Запуск фаззинга

К этому моменту готовы:
- сервер (объект фаззинга) и мод. клиент;
- нач. сценарий их взаимодействия;
- нач. образец вх. данных (привязан к нач. сценарию);
- вспомогательные пользовательские плагины.

Запуск фаззинга:
```shell
$ ./fuzz.sh
```

Опции фаззера:

* `--start <num>` - число fuzz-процессов (экземпляры фаззинга);
* `--eat-cores <num>` - число eat-процессов (процессы доп. анализа);
* `--dse-cores <num>` - число dse-процессов (динамическое символьное выполнение), в данном режиме не поддерживается;
* `--wait-next-instance <milliseconds>` - задержка (в миллисекундах) между запусками fuzz/eat-процессов (для избежения конфликтов);
* `--configurator-script <path>` - скрипт модификации опций (для избежения конфликтов);
* `--eat-sync` - включить синхронизацию через инструмент расширенного анализа;
* `-T <type>` - тип транспорта (`ModClient/ModClientUdp` - в случае фаззинга сервера мод. клиентом, `ModServer/ModServerUdp` - в обратном случае);
* `--mod-client <path>` - Python3-скрипт запуска мод. клиента;
* `--port <num/__free_port>` - порт, на который отправляются мутированные данные (если указан спецификатор `__free_port`, то фаззер подставит вместо него свободный порт);
* `--delay <milliseconds>` - активное ожидание, пока tcp-порт будет занят, перед отправкой мутированных данных (в миллисекундах);
* `--pstate <detectors>` - детекторы состояний (см. "ISP-Circea" в Рук-ве пользователя");
* `-I <type>` - тип инструментации;
* `--redirect-stdin-off` - не перенаправлять stdin в /dev/null (по умолчанию перенаправляется);
* `-i <path>` - путь до директории с начальными образцами входных данных;
* `-o <path>` - путь к выходной директории с результатами фаззинга;
* `-t <milliseconds>` - таймаут на запуск приложения (в миллисекундах).

2. Мониторинг фаззинга.

Запуск ui (в другом терминале):
```shell
$ ./ui.sh
```

Число крешей отображается в поле `unique_crashes` в правом верхнем окне.
Их поиск может занять длительное время. Пример креша можно найти в папке `crashes`.

# 8. Анализ результатов

Образцы входных данных, приводящие к крешам - сохраняются в директории `out/EAT_OUT/crashes`.

## 8.1. Воспроизведение крешей

Пример воспроизведения креша:

1) Запустить сервер:
```shell
$ ./server.sh
```

2) Запустить клиент (с указанием образца вх. данных):
```shell
$ REPLAY_FILE=out/EAT_OUT/crashes/id_crash_000000 ./client.sh
```

Сервер завершится с сигналом `SigSegv`.

## 8.2. Отчёты о покрытии

Просмотр отчётов:

1. Исходный код.
```shell
$ ./show_lcov.sh
```
2. MC/DC.
```shell
$ ./show_mcdc.sh
```
3. Бинарный код.
Файл `out/EAT_OUT/results/lighthouse_total` можно загрузить, например, в IDA или в Binary Ninja с использованием плагина [lighthouse](https://github.com/gaasedelen/lighthouse).
